## 1. Context

On 2026-04-17 the v0.27.0 production deploy failed with:

> «TooManyResourcesInStack» Number of resources in stack 'SwarmApi-prod': **511 is greater than allowed maximum of 500**

This is a **hard CloudFormation limit**, not a tunable. It forces us to look past the immediate unblock and ask whether the architecture can support continued feature growth, or whether the current pattern will re-hit the wall inside a quarter.

The audit in `docs/infra/stack-audit-2026-04-17.md` established:

- **AdminApi = 377 / 496 resources** (76 % of staging stack) — driven by 109 routes + per-route Lambda permission + per-route integration.
- **SharedHandlers = 111 resources** — async workers, queues, alarms, log groups.
- **Growth per new endpoint ≈ 3 CFN resources.** Linear in features, sublinear on payoff.
- The 500-resource ceiling is a symptom; the pattern (one Lambda per endpoint + one Route per method + per-route Permission) is the cause.

A tactical −42 resources (PR #1359 — drop redundant explicit OPTIONS routes, rely on HttpApi native CORS) has been merged and restores ~30-resource headroom. That buys weeks, not quarters.

## 2. Problem

Three coupled problems, in priority order:

1. **Immediate (P0)** — resource-limit budget. Any feature PR that adds one handler costs 3+ resources; at current velocity the next 10 endpoints will re-hit 500.
2. **Durable (P1)** — per-endpoint Lambda packaging scales linearly in CFN resources, deploy time, `cdk synth` time, `cdk diff` noise, and alarm sprawl. Deploys now take 10+ minutes and a single handler change re-synthesizes ~30 Lambda assets.
3. **Quality-of-life (P2)** — monolithic stack means one `cdk deploy` rolls or rolls-back the whole surface. Blast radius is "everything or nothing". Reasoning about change impact is hard.

## 3. Decision

**Adopt a router-Lambda pattern for AdminApi.** Collapse the 109 per-endpoint routes + per-endpoint Lambdas into a small number of **domain router Lambdas** fronted by `{proxy+}` catch-all routes. Handlers become internal functions dispatched by the router; CFN sees 1 Lambda + 1 Route + 1 Permission per domain instead of per endpoint.

Supporting sub-decisions:

- **Router engine: Hono.** Small (~15 KB), no reflection, first-class Lambda runtime, matches Node 20. No veto received.
- **Domain granularity: 3 routers + N kill-switch lambdas.** Exact split decided during Phase 2 prototype; current sketch is `core` / `billing` / `legal`. Webhook-signing lambdas (Stripe, Telegram, Privy, Twitter OAuth callback) stay standalone.
- **SharedHandlers stays as-is for now.** Async workers don't suffer the routing-surface problem. Nested-stack consideration deferred to Phase 5 after router lands — we may not need it.
- **Stack topology unchanged during Phases 0–3.** Stack-split decisions deferred to Phase 5 so we can decide against fresh numbers.
- **Hard cutover per domain.** QA user count is small (< 10 engineers + QA); no shadow routes. Bake period + parity fixtures substitute for traffic-mirroring safety net.

### 4.3 Observability & tracing

- Router Lambdas must preserve **correlation-ID propagation** (webhook → SQS → processor → sender). Correlation ID is injected into the router's request context and forwarded to internal dispatch.
- Per-handler structured logs continue (log-group-per-router, module-tagged log lines). No reduction in observability granularity — the dispatch name replaces function name as the log dimension.
- Alarms continue per-router (not per-endpoint). Acceptable: we trade endpoint-scoped alarming for router-scoped alarming; existing per-domain SQS alarms stay.
- X-Ray / active tracing stays enabled.

### 4.4 Cold start & packaging

- Single-router bundle size estimated 30–50 MB unzipped (vs. ~4 MB per-handler today). Within Lambda's 250 MB S3 upload limit.
- Cold-start budget: **p99 ≤ 1.5 s** for router Lambda. If exceeded, split the router further.
- Shared dependency layer stays as-is (already used by current handlers).

## 5. Non-goals

Explicitly **out** of this overhaul:

- Data-layer redesign. Single-table DynamoDB (admin table + GSI1) stays.
- Behavioral changes to handler logic. Parity-only migration.
- Frontend / admin-ui / profile-page changes.
- Runtime change (stay on Lambda Node 20).
- Auth provider migration (Privy / wallet / invite codes stay).
- Multi-region, DR, active-active.
- Secrets rotation, KMS customer-managed keys.
- Billing / entitlements logic (Web2 Floor + Web3 Ceiling model stays).
- SharedHandlers internal changes (those are async-workload concerns; deferred to Phase 4 review).

### 6.2 Phase 1 — tactical unblock (landed)

PR #1359 (`fix/issue-1355-remove-options-routes`) removed per-route `HttpMethod.OPTIONS` entries. API Gateway HttpApi's native `CorsConfiguration` handles preflight without per-route resources. Effect: −42 resources. Prod 511 → ~469.

Risk: any client relying on Lambda-handled OPTIONS (e.g., consuming custom response headers from the Lambda CORS code path) breaks. Mitigation: `allowedOrigins` is exact-match (no wildcards); all documented cross-origin flows go through same-origin CloudFront proxy.

### 6.3 Phase 2 — router prototype

**Scope:** one router Lambda handling the **auth** domain. Auth is chosen because:

- Relatively self-contained (no deep cross-domain calls).
- Medium traffic — exercises cold start and dispatch realistically.
- Includes both `POST` (challenge/verify) and `GET` (me/gate-status) so both paths are covered.
- Excludes webhook-signed endpoints (Stripe/Telegram/Privy webhook) which stay separate.

**Deliverables:**

1. `packages/admin-api/src/routers/auth-router.ts` — Hono app that dispatches to existing handlers without modifying them. Handler modules become pure request/response functions; router composes auth middleware + CORS + tracing.
2. `packages/infra/src/constructs/admin-api.ts` edit: single `ANY /auth/{proxy+}` route → auth-router Lambda. Remove the ~8 per-endpoint auth routes + lambdas.
3. `packages/admin-api/src/routers/__tests__/auth-parity.test.ts` — recorded fixtures from staging (sanitized) replayed against the router.
4. Staging deploy + 24 h bake.

**Exit criteria:**

- Parity fixtures pass.
- Staging error rate + p95 latency within ±10 % of pre-migration baseline.
- Resource count drop measured in synth and recorded.

### 6.4 Phase 3 — migration waves

One PR per wave, merged + baked independently:

1. **Wave A — core router.** Routes: `/chat`, `/shared-chat/*`, `/avatars/*`, `/issues/*`, `/jobs/*`, `/prompt-preview`, `/api/profile/*`, `/api/leaderboard`, `/transcribe`, `/health`.
2. **Wave B — billing router.** Routes: `/billing/checkout`, `/billing/portal`. Stripe webhook stays standalone.
3. **Wave C — legal router.** Routes: `/consent*`, `/dsar/*`. Kept isolated for compliance review boundary.
4. **Wave D — OpenAI compat router.** Routes: `/v1/chat/completions`, `/v1/models`. May or may not merge into core — decided in Phase 2 retro.

Each wave follows the same cut-over playbook:

```
1. Add router Lambda and new {proxy+} route (new resources, old ones still in place).
2. CI + staging deploy.
3. Run parity fixture suite.
4. Cut over in CDK: remove old per-endpoint routes + lambdas in the same PR.
5. Deploy to staging; 24h bake.
6. Deploy to prod.
7. Delete dead handler stubs (OPTIONS handling, etc.) in a follow-up.
```

### 6.5 Phase 4 — async-path review

After Phase 3, open a scoped review of SharedHandlers:

- Are all 14 SQS queues still needed? (some may be orphans from deprecated features)
- EventBridge rules: 7 active — each justified?
- DLQ pattern: is auto-redrive working as designed (#643)?
- Custom log-group pattern (`LogGroupWithRetention`): 12 × 2 = 24 custom resources. Replace with native `AWS::Logs::LogGroup` via CFN import? Worth ~12 resources saved.
- Correlation-ID gaps in fan-out paths.

Deliverable: review doc + targeted PRs per finding.

### 6.6 Phase 5 — stack-split re-evaluation

After Phase 3 delivers the projected resource drop, re-measure:

- If top-level `SwarmApi-<env>` is < 250 with > 50 % headroom, **close #1353 as no-longer-needed.** Monolithic stack is fine.
- If still tight, execute the SharedHandlers nested-stack plan properly: resource-import migration (not destroy-and-recreate), maintenance window, Telegram webhook URL re-registration runbook.

## 8. Risks

### R1. Parity regression during cutover
**Likelihood:** medium. Handler logic is extracted, not rewritten, but router middleware ordering can introduce subtle differences (auth evaluated earlier/later, error-response shapes).
**Impact:** high — regression surface is every endpoint in the domain.
**Mitigation:** recorded-fixture parity test per endpoint; fail the PR if any byte diff in response headers / body beyond whitelisted fields (timestamps, trace IDs).

### R2. Router cold start > 1.5 s p99
**Likelihood:** medium. Bundle size scales with handler count.
**Impact:** medium — admin UI perceived slowness; no SLO breach in current KPIs.
**Mitigation:** provisioned concurrency if needed; split router further if persistent; profile with AWS Lambda Power Tuner before cut-over.

### R3. Single router Lambda = widened blast radius
**Likelihood:** medium. A bad deploy of `core-router` takes out the whole core domain at once.
**Impact:** high — full-domain outage vs. per-endpoint outage today.
**Mitigation:** canary deploy (first 10 % of traffic) via weighted alias; auto-rollback on 5xx breach; kill-switch webhook lambdas stay isolated.

### R4. Lost per-endpoint alarms
**Likelihood:** high — current alarms are per-Lambda-function.
**Impact:** low-to-medium — alarms still fire, they just attribute at router granularity.
**Mitigation:** add in-router metric emission with `endpoint` dimension so CloudWatch still gets per-endpoint signal; update dashboards.

### R5. Correlation-ID drop
**Likelihood:** low if implemented correctly, high if forgotten.
**Impact:** high — existing incident tooling relies on correlation IDs through webhook → SQS → processor → sender. Router must propagate.
**Mitigation:** middleware in router reads X-Correlation-ID header or generates one, attaches to request context, adds to all SQS sends. Explicit test.

### R6. CFN resource-import fragility (Phase 5 nested stack, if needed)
**Likelihood:** medium — CFN stack resource import is well-documented but finicky with retained resources (log groups, queues).
**Impact:** high — messed-up import orphans resources or requires manual AWS fixes.
**Mitigation:** Phase 5 is opt-in; only execute if Phase 3 doesn't free enough. If executed, run in staging first with full dry-run, rehearse rollback, schedule maintenance window.

### R7. CORS regression from Phase 1 (OPTIONS removal)
**Likelihood:** low — native HttpApi CORS tested under load; exact-origin list; same-origin proxy for bot domains.
**Impact:** medium — cross-origin API calls would fail at preflight.
**Mitigation:** monitored — PR #1359 test plan includes manual preflight smoke from admin UI + bot subdomain on deploy. Rollback available via revert.

### R8. Scope creep into handler rewrites
**Likelihood:** high — router refactor tempts incidental cleanup.
**Impact:** medium — doubles the PR size and review burden; breaks parity bar.
**Mitigation:** PR template enforces "routing change only"; cleanup PRs are follow-ups.

## 9. Kill-switch zones

Do **not** fold these into a shared router in any phase. Each stays as a standalone Lambda with its own route:

| Endpoint | Reason |
|----------|--------|
| `POST /webhook/stripe` | Stripe webhook signature verification + idempotency store. Legal/compliance audit boundary. Consolidating risks signing-key exposure during refactor. |
| `POST /telegram/webhook` (webhooks, SharedHandlers) | Telegram webhook HMAC validation. Re-registration with Telegram is out-of-band; changing the URL is a manual ops step. Must not churn. |
| `POST /auth/privy/verify` | Privy JWT verification — third-party identity dependency, distinct failure mode. Keep isolated until Phase 2 proves router stability. |
| `POST /twitter/oauth/callback` | Twitter OAuth 2 redirect URL registered externally. Changing the integration target risks breaking the connect flow. |
| `POST /consent*` + `POST /dsar/*` | Legal-sensitive; we want the PR / change log for these endpoints to be narrowly scoped and reviewable. Legal router (Phase 3 Wave C) keeps them together but separate from `core`. |
| DLQ processor, MessageProcessor, ResponseSender (SharedHandlers) | Async event-path; routing refactor does not apply. Phase 4 review handles these. |

## Appendix A: How to synth prod-shaped template locally

`stackHash=prod00` in `cdk.context.json` fails the hex-regex in `bin/swarm.ts:135`. CI passes it explicitly. For local work, use staging (identical topology) or pass a valid 6-hex `stackHash`:

```bash
cd packages/infra
CDK_DEFAULT_ACCOUNT=022118847419 CDK_DEFAULT_REGION=us-east-1 \
  npx cdk synth --app "npx tsx bin/swarm.ts" \
  -c environment=staging \
  -c useExistingResources=true \
  -c skipDomainAliases=true \
  -c stackHash= \
  --quiet \
  SwarmApi-staging
```

## Appendix C: Why not Option X

- **Why not split AdminApi into its own top-level CFN stack (original #1354)?** Bigger blast radius at deploy (two stacks to coordinate), doesn't address per-endpoint pattern, defers the real fix. Resource cap returns in months.
- **Why not nested stack for SharedHandlers now?** Logical-ID churn in CFN ⇒ recreate SQS queues, lambdas, event-source mappings. Requires resource-import migration + maintenance window + Telegram webhook re-registration. Worth doing only if router work doesn't free enough; defer to Phase 5.
- **Why not container-image Lambda?** 10 GB image limit sidesteps resource count only trivially; cold-start worse; doesn't change the CFN route surface.
- **Why not AppSync / REST API Gateway?** Change of API style ≠ resource-count fix. Out of scope.
- **Why not break the stack into many micro-stacks?** Eight 100-resource stacks still have 8× the coordination cost. Router + single stack is strictly simpler when the routing change alone fixes the growth pattern.
