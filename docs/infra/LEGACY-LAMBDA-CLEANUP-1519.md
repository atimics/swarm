# Legacy Lambda Cleanup (#1519)

## Issue Summary

Production is running duplicate EventBridge-scheduled Lambda functions:

- **Legacy (orphaned)**: `swarm-prod-X` without `-split` suffix  (25MB, stale)
- **Authoritative (current)**: `swarm-prod-split-X` with `-split` suffix (15MB, active)

Both are processing identical workloads, causing **2× Lambda invocation cost** and **2× API calls**.

## Root Cause

The nested-stack migration (PR #1427, later reverted in #1434) deployed new `-split` versions of all handlers. When the migration was reverted:

1. Current CDK code kept using the `-split` suffix for prod/staging handlers
2. Old EventBridge rules targeting non-split Lambdas were orphaned in AWS
3. Both the old and new Lambdas are scheduled and running concurrently

This drift is not caught by CDK because:
- CDK only manages the authoritative `-split` versions
- The non-split Lambdas are legacy remnants in AWS that CDK doesn't see
- Removing them via CDK would cause CloudFormation conflicts (can't delete what it doesn't manage)

## Affected Handlers

The following handlers have EventBridge schedules and need cleanup:

| Lambda | Schedule | Artifacts |
|--------|----------|-----------|
| `twitter-mention-poller` | Every 1 minute | Legacy: `swarm-prod-twitter-mention-poller` |
| `autonomous-tweet-poster` | Every 1 hour | Legacy: `swarm-prod-autonomous-tweet-poster` |
| `platform-heartbeat` | Every 15 minutes | Legacy: `swarm-prod-platform-heartbeat` |
| `station-agent-runner` | Every 1 hour | Legacy: `swarm-prod-station-agent-runner` |
| `dlq-processor` | Every 15 minutes | Legacy: `swarm-prod-dlq-processor` |

- `telegram-webhook`

## Remediation: Manual AWS Cleanup

**Why manual?** These are remnants outside CDK's management scope. Automating removal risks CloudFormation conflicts.

### Prerequisites

- AWS CLI configured with `--profile prod` access
- Permission to disable EventBridge rules and inspect Lambda functions
- Access to CloudWatch to verify activity

### Step 1: Identify Legacy Lambdas and Their Rules

```bash
# List all `swarm-prod-*` (non-split) Lambdas
aws lambda list-functions --region us-east-1 --profile prod \
  --query "Functions[?FunctionName | starts_with(@, 'swarm-prod-') && !contains(FunctionName, 'split')] | [].[FunctionName, LastModified, CodeSize]" \
  --output table

# Expected output shows stale last-modified dates and/or 0 bytes
```

### Step 2: Verify Legacy Lambdas Have No Recent Activity

For each legacy Lambda (e.g., `swarm-prod-twitter-mention-poller`):

```bash
# Check CloudWatch log group age (should be stale or empty)
aws logs describe-log-groups --region us-east-1 --profile prod \
  --log-group-name-prefix "/aws/lambda/swarm-prod-twitter-mention-poller"

# Expected: log group exists but /aws/lambda/swarm-prod-split-twitter-mention-poller is active
# Check which log group has recent entries

aws logs tail /aws/lambda/swarm-prod-twitter-mention-poller --region us-east-1 --profile prod --since 1h --follow
# Should show no logs (or activity stopped days ago)
```

### Step 3: Identify EventBridge Rules Targeting Legacy Lambdas

```bash
# List all EventBridge rules
aws events list-rules --region us-east-1 --profile prod --query "Rules[].Name" --output text

# For the traffic poller, check rule targets
aws events list-targets-by-rule --rule "TwitterMentionPollSchedule" --region us-east-1 --profile prod

# Should show the split version: Arn contains "swarm-prod-split-twitter-mention-poller"

# If there's a separate rule like "TwitterMentionPoll-legacy", list its targets:
# aws events list-targets-by-rule --rule "TwitterMentionPoll-legacy" --profile prod
```

### Step 4: Disable Legacy EventBridge Rules (Staged)

**CAUTION**: Disable before deleting to ensure no invocations during cleanup.

```bash
# For each legacy Lambda, find its EventBridge rule name
# Common pattern: Rules created automatically by CDK may have autogenerated names

# Get all rules (filter manually for those targeting non-split Lambdas):
aws events list-targets-by-rule --rule "<RuleName>" --region us-east-1 --profile prod

# If a rule targets a legacy Lambda (non-split), disable it:
aws events disable-rule --name "<RuleName>" --region us-east-1 --profile prod

# Example:
# aws events disable-rule --name "TwitterMentionPoll-legacy" --region us-east-1 --profile prod
```

**Expected Outcome**: After disabling, CloudWatch metrics should show only the `-split` version receiving invocations.

### Step 5: Verify Duplicate Invocations Have Stopped

```bash
# Monitor CloudWatch metrics for 1 minute window
aws cloudwatch get-metric-statistics \
  --namespace AWS/Lambda \
  --metric-name Invocations \
  --start-time $(date -u -d '10 minutes ago' +%Y-%m-%dT%H:%M:%S) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%S) \
  --period 60 \
  --statistics Sum \
  --region us-east-1 \
  --profile prod \
  --dimensions Name=FunctionName,Value=swarm-prod-split-twitter-mention-poller \
             Name=FunctionName,Value=swarm-prod-twitter-mention-poller

# After disabling legacy rule, only the "swarm-prod-split-*" should show non-zero invocations
```

## Rollback

If issues arise after disabling legacy rules:

```bash
# Re-enable the EventBridge rule
aws events enable-rule --name "<RuleName>" --region us-east-1 --profile prod
```

This will resume invocations to the legacy Lambda (temporary safety net while debugging).
