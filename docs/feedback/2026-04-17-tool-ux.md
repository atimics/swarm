## 1. Tool error messages are opaque (P-high candidate)

**Symptom.** Integration tool calls (Twitter post, Telegram send, Discord webhook, Replicate generate) return thin `{ success: false }` responses. Avatar can't distinguish rate-limit from auth-expired from payload-rejected, so user feedback is useless.

**Desired.** Propagate actionable detail from the adapter up through the tool response:
- `"Tweet too long: 285 / 280 chars"`
- `"Telegram webhook unreachable (502) — last healthy 12m ago"`
- `"Replicate job rejected: reference image 413 Payload Too Large"`

**Likely scope.** `core/` adapters + tool response envelope. `admin-api` shape change. Avatar prompt update to surface reasons.

**Related.** Covers outside-feedback #6 and inside-feedback #1.

## 2. Pre-action credit / energy check in tool prompts (P-high candidate)

**Symptom.** Avatar discovers credit exhaustion only *after* attempting generate_image / generate_video, producing a confusing failure. `get_tool_credits` exists but isn't called proactively.

**Desired.** Tool wrappers surface remaining balance before invocation and ask for confirmation when the action would exhaust or dip below a threshold: `"You have 2 video credits. This generation uses 2. Proceed?"`

**Likely scope.** Tool-use prompt layer (`core/tools/*`) or admin-api tool envelope. Possibly an `assert_credits(n)` pre-flight call injected by the avatar runtime.

**Overlaps.** #1270 (admin-UI credit dashboard) is UI-side; this is the avatar-prompt-side counterpart.

**Related.** Covers outside #3, inside #5.

## 3. Async generation continuation UX is clunky (P-medium candidate)

**Symptom.** `generate_image` / `generate_video` return a job ID. No proactive notification on completion; avatar either polls or waits for a continuation message. From the user's POV this reads as "did it work? is it still running?"

**Desired.** Either:
- Auto-poll + push notification back into the conversation when the job finishes, with a "use this in your next action" affordance; or
- First-class continuation primitive the avatar can reference by short handle instead of raw job ID.

**Likely scope.** `core/` tool runtime + admin-api event fan-out. Possibly an SQS/EventBridge bridge back to the chat session.

**Related.** Covers outside #4, inside #4.

## 4. Reference image management is under-discovered (P-low candidate)

**Symptom.** `set_character_reference` exists but most users never find it. Avatars with turnaround sheets render far more consistently, but the capability is invisible.

**Desired.** Guided CTA during onboarding — "Want consistent designs across generations? Upload a reference." — plus a proactive suggestion when the avatar notices inconsistent generations without a reference set.

**Likely scope.** Onboarding flow in `admin-ui`; avatar system-prompt cue.

**Related.** Covers outside #5, inside #6.

---

## 5. Onboarding doesn't explain persistence boundaries (P-medium candidate)

**Symptom.** Users discover by accident that conversational memory doesn't carry between sessions while profile / persona / galleries do persist. They're confused when the avatar doesn't remember them next session.

**Desired.** Explicit welcome message (and a one-liner reminder on session start if memory is ephemeral for that tier) distinguishing what persists vs. resets.

**Likely scope.** Onboarding wizard copy + session-start prompt injection. Small.

**Related.** Covers outside #1.

---

## 6. Integration setup wizard + unified status view (P-medium candidate)

**Symptom.** Many integration options (Telegram, Twitter, Discord, Replicate, OpenAI, Anthropic, OpenRouter, Helius). First-time users don't know where to start; `configure_integration` vs `request_secret` vs `store_secret` is guesswork. `get_integration_status` is per-integration; no dashboard.

**Desired.**
- First-run chat-native wizard: "Social? Bot platform? Media generation?" → recommends minimum-viable config.
- `list_integration_status` returning all integrations at once, with health + missing-secret indicators.

**Likely scope.** `admin-api` aggregator + avatar prompt. Follows the chat-first design philosophy (no settings pages).

**Related.** Covers outside #2, inside #3.

---

## 7. Gallery browse/pick primitive (P-medium candidate)

**Symptom.** Avatar has to track image IDs as strings and call `list_gallery` repeatedly. Typos break `twitter_post` and similar.

**Desired.** Structured selector — a tool or prompt affordance where the avatar picks by index/label from a short list returned inline, instead of round-tripping IDs through conversation.

**Likely scope.** Core tools; possibly a new `pick_from_gallery` tool that returns a short-lived handle.

**Related.** Covers inside #2.

---

## 8. Avatar status (active / draft / paused) clarity (P-low candidate)

**Symptom.** `set_avatar_status` exists but the transitions and what each state means (inbound traffic? scheduled posts? memory writes?) aren't surfaced. Users don't know their own setup state.

**Desired.** Admin-UI surfaces current status prominently; docs define each state's side effects; `get_avatar_status` returns a one-line human summary.

**Likely scope.** `admin-ui` + small avatar-service doc.

**Related.** Covers outside #7.

---

## 9. Profile preview / draft mode before publishing (P-low candidate)

**Symptom.** `update_my_profile` publishes immediately; avatar can't show the user what the change will look like before it goes live.

**Desired.** Preview mode that renders the proposed profile card inline before committing. Cancel / commit actions.

**Likely scope.** `admin-ui` + admin-api staging field on profile.

**Related.** Covers inside #7.
