# Dynamic Context & Prompt Reduction RFC

Status: Draft
Author: Internal
Date: 2026-01-25

> **Sequencing Note**: This RFC is scheduled for **post-M1** (after paid Telegram MVP ships).
> It builds on completed work from MEMORY.md Phases 2-4 and integrates with TOOL-COMPOSITION-OBSERVABILITY-RFC.
> See [Integrated Roadmap](#integrated-roadmap) below for sequencing.

## Summary

This RFC proposes a significant restructuring of how avatar system prompts are built, replacing prescriptive instruction blocks with dynamic, contextual information. The goal is to reduce prompt size by ~50% while improving relevance and cross-platform coherence.

**Key changes:**
1. **Channel summaries**: Per-channel cached summaries generated by Haiku, providing cross-platform context (supersedes TOOL-COMPOSITION-RFC P2 "cross-platform context")
2. **Dynamic toolset activation**: LLM explicitly activates toolsets on demand, with auto-routing suggestions
3. **Pinned memories**: Avatar self-curates important facts that are always included in context
4. **Prompt reduction**: Remove prescriptive capability sections, trust tool schemas

## Prerequisites

This RFC depends on:
- **M1 complete**: Paid Telegram MVP, entitlements, memory retention policies
- **MEMORY.md Phase 2**: Scheduled consolidation (memories decay/promote properly)
- **MEMORY.md Phase 3**: Identity evolution ("I am becoming..." statements exist)
- **SEMANTIC-MEMORY-DESIGN**: Embedding backfill complete, semantic search hardened
- **TOOL-COMPOSITION-RFC P0**: system_status and my_activity tools available

## Problem Statement

### Current prompt structure wastes tokens on irrelevant instructions

The current system prompt includes detailed capability sections (~300-1000 tokens) that describe *how* to use tools:

```markdown
## Media Generation
You have media generation capabilities:
- generate_image: Generate images (20 credits max, refills 10/hour)
- generate_video: Generate videos (3 credits max, refills 1/hour)
- Always ask for confirmation before generating...
```

This is wasteful because:
- Tool schemas already describe tool capabilities
- Instructions are only relevant when actually using media tools
- The same instructions repeat every message, regardless of context

### Cross-platform context is missing

When handling a Telegram message, the avatar has no awareness of:
- What it posted to Twitter 2 hours ago
- What it discussed in the admin UI yesterday
- Active conversations on other platforms

Each handler loads the same memories but context is siloed by platform.

### Memory injection is not contextual

Current memory injection appends all relevant memories to the prompt. There's no:
- **Pinned memories**: Facts the avatar considers critical
- **Contextual retrieval**: Memories relevant to *this specific message*
- **Relationship memory**: Automatic inclusion of sender context

### All tools are always visible

Even with the existing tool router (which limits to 3 toolsets), the filtering is basic keyword matching. The LLM can't explicitly request additional tools when needed.

## Proposed Design

### 1. Channel Summary Service

Each channel (Telegram chat, Twitter timeline, Discord channel) gets its own cached summary, generated by Haiku with avatar context.

#### Schema

```typescript
// DynamoDB: pk=SUMMARY#{avatarId}, sk=CHANNEL#{platform}#{channelId}
interface ChannelSummary {
  pk: string;                    // SUMMARY#{avatarId}
  sk: string;                    // CHANNEL#{platform}#{channelId}
  avatarId: string;
  platform: string;              // 'telegram' | 'twitter' | 'discord' | 'admin-ui'
  channelId: string;             // Chat ID, timeline, channel ID
  channelName?: string;          // Human-readable name
  summary: string;               // Haiku-generated, in avatar's voice
  messagesSinceUpdate: number;   // Messages since last summary
  lastMessageAt: number;         // Timestamp of last message
  generatedAt: number;           // When summary was generated
  expiresAt: number;             // Cache expiry
  ttl: number;                   // DynamoDB TTL (expiresAt + buffer)
}
```

#### Summary Generation Flow

```
Avatar posts/receives message on channel
        │
        ▼
Log activity event (lightweight, always)
        │
        ▼
Is this a DIFFERENT channel than current?
        │ (yes)
        ▼
Check: should we update that channel's summary?
  - Last summary > 15 min ago?
  - 5+ messages since last update?
        │ (yes to either)
        ▼
Enqueue summary job (SQS, async)
        │
        ▼
Worker picks up job
        │
        ▼
Load: avatar persona (condensed), recent messages on channel
        │
        ▼
Haiku generates summary (in avatar's voice, 2-3 sentences)
        │
        ▼
Cache summary (TTL: 1 hour)
```

#### Rate Limiting

| Limit | Value | Scope |
|-------|-------|-------|
| Per-channel cooldown | 15 minutes | Channel |
| Per-avatar hourly limit | 10 summaries | Avatar |
| Max channels tracked | 20 | Avatar |

#### Summary Generation Prompt

```markdown
You are summarizing recent activity for {avatar.name} on {platform} in {channelName}.

Avatar persona (condensed):
{avatar.personaEssence}

Recent messages (last 10):
{messages}

Write a 2-3 sentence summary from the avatar's perspective, using first person.
Focus on: topic of conversation, key participants, any decisions or outcomes.
Keep it brief and natural, matching the avatar's voice.
```

#### Injection into System Prompt

```markdown
## Recent Context

**This channel** ({platform} {channelName}):
{currentChannelSummary or "New conversation"}

**Other active channels:**
- Twitter (2h ago): Posted thread about AI art, got 12 likes
- Discord #general (30m ago): Discussing wallet setup with @user
```

### 2. Dynamic Toolset Activation

Replace implicit tool filtering with explicit activation. The LLM sees a small set of active tools plus a list of available toolsets it can request.

#### Toolset State

```typescript
// Stored in conversation state (DynamoDB or handler memory)
interface ToolsetState {
  avatarId: string;
  conversationId: string;
  activeToolsets: ToolsetId[];       // Currently active (tools visible)
  suggestedToolsets: ToolsetId[];    // Auto-detected from message
  availableToolsets: ToolsetId[];    // Can be activated
}
```

#### Prompt Format

```markdown
## Tools

**Active:** core, memory (12 tools)
**Suggested:** media, twitter (detected from your message)
**Available:** wallet, voice, telegram, discord, property, nft, diagnostics

Call `activate_toolset("name")` to enable additional tools.
```

#### Auto-Routing Enhancement

The existing `tool-router.ts` continues to work but now:
1. Scores toolsets based on message content (existing logic)
2. Top-scoring toolsets become "suggested" (shown in prompt)
3. Only `core` + `memory` (if enabled) are active by default
4. LLM must explicitly activate suggested or available toolsets

This reduces initial tool definitions from ~30-60 to ~15-20.

### 3. Pinned Memories

Avatar self-curates important facts that bypass normal retrieval and are always included.

#### Schema Extension

```typescript
// Extend AvatarMemory
interface AvatarMemory {
  // ... existing fields
  pinned?: boolean;              // Is this memory pinned?
  pinnedAt?: number;             // When it was pinned
  pinnedReason?: string;         // Why (for avatar reference)
  pinnedOrder?: number;          // Display order (1-10)
}
```

#### Separate Index for Fast Retrieval

```typescript
// GSI: PINNED#{avatarId} for fast lookup
// Or separate lightweight table

interface PinnedMemoryIndex {
  pk: string;                    // PINNED#{avatarId}
  sk: string;                    // ORDER#{paddedOrder}#{memoryId}
  memoryId: string;
  content: string;               // Denormalized for fast load
  pinnedAt: number;
  pinnedReason?: string;
}
```

#### Prompt Injection

Pinned memories appear first, before contextual memories:

```markdown
## Important Facts (pinned)
- I am Rati's avatar, deployed on AWS Swarm
- I never share private keys or seed phrases

## Contextual Memory

- Last conversation with @sender was about NFT art
```

### 4. Prompt Reduction

Remove prescriptive sections, trust tool schemas, condense where possible.

#### Sections to Remove

| Section | Current Tokens | Action |
|---------|----------------|--------|
| Epistemic stance | ~150 | Remove (model knows this) |
| Operating principles | ~100 | Remove (too prescriptive) |
| Capability sections (13) | ~300-1000 | **Remove entirely** |
| Tool guidance | ~100 | Reduce to 1 line |

#### Sections to Add/Modify

| Section | Tokens | Content |
|---------|--------|---------|
| Core identity | ~100 | Name, condensed persona, current state |
| Pinned memories | ~150 | Self-curated important facts |
| Channel context | ~100 | This channel + other channel summaries |
| Contextual memories | ~100 | RAG-retrieved for this message |
| Toolset status | ~30 | Active/suggested/available |
| Platform hint | ~20 | "On Twitter: 280 chars" |

#### Before vs After

**Before (~1500 tokens):**
```markdown
## Identity
You are {name}. {description}

## Persona
{persona - 200+ tokens}

## Epistemic Stance
{150 tokens of philosophy}

## Operating Principles
{100 tokens of rules}

## Secrets & Integrations
{150 tokens}

## Solana Wallets
{60 tokens}

## Media Generation
{200 tokens}

## Twitter/X Features
{150 tokens}

... (more capability sections)

## Tool Guidance
{100 tokens}

## Platform
{70 tokens}

## Memory Context
{100-300 tokens appended}
```

**After (~600 tokens):**
```markdown
## Who I Am
{name} - {1-sentence essence}
{dream state - 50 tokens}

## Important Facts
- {pinned memory 1}
- {pinned memory 2}
- ...

## Recent Context
**This channel:** {summary}
**Other channels:** {1-line each}

## Relevant Memories
- {contextual memory based on message}
- {relationship memory for sender}

## Tools
Active: core, memory | Suggested: media | Available: [list]

## Platform: {platform}
{1-line constraint, e.g., "Twitter: 280 chars max"}
```

## Migration Plan

> **Important**: This migration assumes M1 and MEMORY.md Phases 2-4 are complete.
> Check prerequisites before starting.

### Pre-flight Checklist
- [ ] M1 P1.3 complete (memory retention controls exist)
- [ ] Entitlement service supports `features.channelSummaries` and `features.pinnedMemories`
- [ ] MEMORY.md Phase 2 complete (consolidation job running)
- [ ] Semantic search hardened (>80% embedding coverage)

### Phase 1: Channel Summaries (1 week)

**Prerequisites**: M1 complete, activity logging infrastructure

1. Create `channel-summary.ts` service
   - Respect entitlements: only generate for paid tier
   - Track Haiku costs per avatar
2. Create summary worker Lambda + SQS queue
3. Add activity logging to handlers (lightweight events)
4. Inject channel context into prompt (additive, no removal yet)
5. Test with 1-2 paid avatars

**Rollback**: Feature flag `ENABLE_CHANNEL_SUMMARIES=false`

### Phase 2: Pinned Memories (1 week)

**Prerequisites**: Consolidation running, retention policies enforced

1. Add `pinned` field to memory schema
   - Pinned memories respect retention TTL
   - Free tier: max 3 pinned, paid tier: max 10
2. Implement pin/unpin tools
   - Tools check entitlement before allowing pin
3. Modify `getMemoryContext()` to include pinned first
4. Modify `getPinnedMemories()` to skip expired memories
5. Test self-curation behavior

**Rollback**: `pinned` field ignored if feature flag off

### Phase 3: Toolset Activation (1 week)

**Prerequisites**: None (can run in parallel with Phase 2)

1. Implement `activate_toolset` / `deactivate_toolset` / `list_toolsets`
2. Modify tool filtering to respect activation state
3. Update prompt to show active/suggested/available
4. Test with media-heavy and multi-tool workflows
5. Consider: should `system_status` (from TOOL-COMPOSITION-RFC) be auto-activated?

**Rollback**: Fall back to current auto-routing behavior

### Phase 4: Prompt Reduction (1-2 weeks)

**Prerequisites**: Phases 1-3 tested and stable

1. Create `prompt-builder-v2.ts` with new structure
   - Start with paid tier only (has all features)
   - Free tier gets minimal prompt (identity + platform hint only)
2. A/B test: old prompt vs new prompt
   - Metrics: token usage, response quality, tool success rate
   - Qualitative: avatar voice consistency
3. Gradual rollout via feature flag per avatar
4. Deprecate `prompt-builder.ts` after 2 weeks stable

**Rollback**: Feature flag `USE_PROMPT_V2=false`

## Open Questions

1. **Summary staleness**: Is 15-min cooldown too aggressive? Should we also trigger on message count?
2. **Toolset persistence**: Should activated toolsets persist across conversations or reset each time?
3. **Pinned memory admin override**: Should admin be able to force-pin certain facts?
4. **Persona condensation**: How do we condense a long persona without losing voice? Use Haiku?
5. **Gradual rollout**: Feature flag per avatar, or global switch?

## Security Considerations

- Channel summaries should not include sensitive content (filter before summarizing)
- Pinned memories are avatar-controlled; admin cannot read them without avatar consent
- Toolset activation respects existing category/platform filtering

## Entitlement Considerations

Channel summaries and pinned memories have cost implications:

| Feature | Free Tier | Paid Tier |
|---------|-----------|-----------|
| Channel summaries | Disabled (no cross-platform context) | Enabled, rate-limited |
| Pinned memories | Max 3 | Max 10 |
| Contextual memory retrieval | Disabled | Enabled with retention limits |
| Haiku summarization calls | N/A | Cost attributed to avatar |

The prompt builder must check entitlements before injecting dynamic context.
